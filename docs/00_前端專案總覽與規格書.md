- **`utils/geometry.js` 為單一真實來源**：`computeDisplayCropGeometry` 與 `DISPLAY_CROP_RATIO` 只定義一次，

- `CameraFlow.jsx` 不再包含任何演算法，只組合 hooks、持有 stage、決定渲染哪支畫面元件。

| 功能 / 元素 | 正式版（prod） | 測試版（test） | 實作位置 |
| ----- | :---: | :---: | ----- |
| 模型偵測框（綠/橘 `<rect>`） | ✗ 不顯示 | ✓ 顯示 | `DetectionOverlay`（測試版才掛載） |
| Raw 感測器尺寸 | ✗ | ✓ | `DebugPanel` |

| 手動快門按鈕 | ✗ **移除，只能自動拍照** | ✓ 保留（不受五項條件限制） | `Viewfinder` |

| 引導框（車牌/輪胎虛線框） | ✓ 顯示 | ✓ 顯示 | `GuideOverlay`（兩版皆有，屬正常引導） |
| 四方位進度 UI | ✓ | ✓ | `PositionCompass` |
| 自動快門 | ✓ | ✓ | `useAutoCapture` |
| Firebase 上傳 | ✓ 寫入正式 collection | ✓ 寫入測試 collection（見 §4.4） | `services/upload.js` |

> 重點：正式版「引導框」保留（那是使用者取景的必要輔助），移除的是「**模型偵測結果框**」與「**除錯數值**」。兩者不同，勿混淆。

### Build 指令

```jsonc
// package.json scripts
"dev":         "vite",                          // 開發（MODE=development，預設帶測試功能）
"build":       "vite build",                    // 正式版（MODE=production → prod）
"build:test":  "vite build --mode test",        // 測試版（MODE=test）
"deploy":      "npm run build && gh-pages -d dist",       // 部署正式版
"deploy:test": "npm run build:test && gh-pages -d dist -b gh-pages-test"  // 部署測試版（另一分支/路徑）
```

> ⚠️ 實作定案：改用 Vite 內建的 `import.meta.env.MODE`（`--mode test` 時為 `"test"`），
> **不需要 `.env.test` 檔**（且遠端寫檔工具禁止寫入 dotenv 檔）。無額外環境變數檔要維護。

## 3. appConfig 版本切換機制

`src/config/appConfig.js` 為唯一判定來源，其他檔案一律 import `IS_TEST_MODE`，不自行讀 env 或網址。

```js
// 兩層機制：
// 1) 建置層（主要）：由 build 指令的 --mode 決定，Vite 打包時把
//    import.meta.env.MODE 靜態替換為字面值 → 正式版的
//    if (IS_TEST_BUILD) {...} 會變成 if (false) 被 tree-shaking 移除，
//    測試專屬元件（DetectionOverlay / DebugPanel / 手動快門 / 校正工具）
//    在正式 bundle 中「完全不存在」。（已用 build 驗證：正式 bundle 不含 DebugPanel
//    的 "Raw:" 字串，測試 bundle 含之。）
// 2) 執行層（次要）：?test 網址參數。方便在「已含測試碼的 build（dev）」
//    上臨時切換呈現。正式 build 除錯碼已移除，?test 無法叫回 —— 刻意設計。

const IS_TEST_BUILD = import.meta.env.MODE === 'test';
const hasTestParam = new URLSearchParams(window.location.search).has('test');

export const IS_TEST_MODE = IS_TEST_BUILD || (import.meta.env.DEV && hasTestParam);
export const APP_MODE = IS_TEST_MODE ? 'test' : 'prod';
export const APP_VERSION = '1.0.0-refactor'; // 進 build 資訊，寫入 Firestore
```

> 為什麼不用純執行期參數：因為那樣正式版 bundle 仍含全部除錯功能，只是預設不顯示，
> 加個 `?test` 就能翻出來 —— 不符合「正式版不出現除錯訊息」的要求。建置層旗標才能真正移除。

## 5. UI/UX 設計規範

### 5.1 設計語言（參考 iRent）

- **配色（`theme.js` tokens）**：主色 iRent 紅（`#E2231A` 類）、深底 `#0E0E10`、卡片 `#1A1A1D`、
  白字 `#FFFFFF`、次要灰 `#9AA0A6`、成功綠 `#00C36E`、警示紅 `#FF4D4F`。
  相機取景畫面維持深色底，資訊頁用白卡片提升商業感。
- **元件**：圓角膠囊按鈕、卡片式表單、明確主/次按鈕層級、一致間距節奏（8pt 基準）。

### 9.2 Firebase 匿名登入健壯化（`config/firebase.js` / `services/upload.js`）

**現象**：`uploadPhotoRecord` 等函式一律 `await authReady`；`authReady` 只有

**自由順序拍攝**：`CameraFlow` 的拍攝狀態從「`positionIndex` + 陣列 `capturedPhotos`

**相機拍攝數次後黑屏、無法恢復（重大修正）**：
- **現象**：測試版連續拍攝約 5 次（含重拍）後，相機畫面卡住變黑，且不會自己恢復；
  快速把瀏覽器切到背景再切回前景，畫面又會恢復。
- **排查過程**：先後排除了「過熱」（tfjs `tensors`/`bytes` 用量五次拍攝下來完全沒有攀升）、
  「感測器解析度不穩」（`raw`/`out` 尺寸五次都一致）——用暫時加在 `DebugPanel`／
  `PreviewScreen` 的 tfjs 記憶體與尺寸診斷（已移除，僅存在於這次排查的過程）確認。
  真正線索是「切背景再切回來就恢復」：真的被系統終止的相機串流不可能靠這動作復活，
  代表 track 其實一直是活的，只是畫面卡住不再繪製。
- **根因**：`<video>` 元素原本包在 `Viewfinder` 裡，而 `Viewfinder` 只在 `stage===SHOOTING`
  時掛載。每拍一張（SHOOTING → PREVIEW → 確認/重拍 → 回 SHOOTING），`<video>` DOM 節點
  就會被整個拆掉再重建一次，同一條 `MediaStream` 反覆重新 `srcObject =` 到不同的
  `<video>` 節點上。部分 WebKit 版本在這種反覆 attach 數次後，會讓新的 `<video>`
  不再實際解碼繪製畫面（track 沒死，只是不畫），且不會自行恢復。
- **修正**：`Viewfinder`（含 `<video>`）在 `SHOOTING` 與 `PREVIEW` 兩階段都保持掛載，
  不再每拍一張就整個拆掉重建；`PreviewScreen` 改為 `position:absolute` + 高 `z-index`
  （`theme.z.preview`）蓋在 Viewfinder 上層的不透明畫面，視覺上不變，但底層 video
  元素與串流全程沒被拆過。`ANALYZING` 之後（分析/複核/下載等）維持原本會卸載相機的
  行為，因為那些是一次性轉場、不是問題根源，沒必要讓相機繼續佔資源。
- **附帶強化**：`useCamera.js` 新增 `track.onended` 監聽——若相機串流真的被系統中止
  （非本地 `stopStream()` 主動呼叫），切回 `CAMERA_STATUS.ERROR`，讓既有「重新嘗試」
  畫面可以恢復，不再永久卡死無法回應。
- ⚠️ **相機解析度需求 `width/height: {ideal:6000}` 維持原樣，勿改**：曾嘗試降到
  `{ideal:1080/1920}` 猜測與過熱有關，結果導致瀏覽器在協商解析度時額外做了一層
  裁切/放大，使 `Viewfinder` 顯示畫面被放大——**違反 §0 的「不要動畫面」原則**，
  已改回 `6000`。過熱／解析度皆已用診斷數據排除，不是這次黑屏問題的成因。

### 9.5 9:16 車牌字元辨識截斷修正（`services/models.js` / `utils/geometry.js`）

**現象**：3:4 拍攝後車牌辨識正常，9:16 拍攝後大多情況只認得出車牌前 1~2 碼。

**根因**：車體定位模型固定把裁切後畫面等比縮進 640×640 正方形推論
（`scale = 640 / max(cropW, cropH)`）。9:16 因裁切高度天生比 3:4 大，縮放係數明顯
較小，車牌在偵測輸入影像中的實際像素比 3:4 少約 25~30%，偵測框的量化誤差相對更大，
容易偏窄、切掉車牌右側字元；`recognizePlateCharacters` 原本只用「框自身寬高的
`CHAR_CROP_PADDING_PERCENT`(10%)」外擴，框越窄補得越少，救不回被切掉的字元。

**修正（兩層）**：
1. `expandBoxByPercent` 新增 `CHAR_CROP_MIN_PADDING_PCT`（畫面百分點的最小外擴保底值，
   與框準確度無關，`Math.max(框自身比例外擴, 保底值)`）。
2. `recognizePlateCharacters` 新增 `templateBox` 參數，裁切車牌時取「偵測框（已外擴）」
   與「該角度／比例人工校正的樣板位置（`guideTemplates`/`carModels` 的 `licensePlate`）」
   的**聯集**當裁切範圍。樣板座標是固定校正值，不受偵測解析度影響，能穩定兜住被
   切掉的字元。`CameraFlow` 在 `takePhoto` 呼叫時傳入目前角度/比例的 `template.licensePlate`。

驗證方式：實測比對 9:16 相關程式碼與更早、確認 9:16 正常的 commit（`365ea9d`）
